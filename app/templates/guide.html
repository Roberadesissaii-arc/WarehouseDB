1 · What this system is

WarehouseDB is the control board for your physical warehouse. It answers two questions at all times: where is every item? and where is every robot?

Each item is one real object on exactly one shelf bay. Each robot is a connected worker that heartbeats to the server, receives numbered work orders, and reports its live status.

Locations are organised in three levels (there is no separate “aisle” row — an aisle is a section):

WAREHOUSE→ AISLE / ZONE→ BAY→ ITEM

Example: Main Site → Aisle A → A-01 → Drive Chain — create the aisle as a section named Aisle A, then add bays A-01, A-02 inside it.

2 · First-time setup (production)

A fresh install starts empty — no sample warehouses or demo robots. Build your real layout before pairing hardware:

  1. Create your account — first visit shows a setup form (one username + password for this server; 8+ characters with a letter and a number). No preset login exists.
  2. Add a warehouse — use the ＋ Warehouse chip on the deck, or Settings → Locations.
  3. Add sections — storage zones inside that warehouse (e.g. Receiving, Aisle A).
  4. Add bays — shelf codes inside each section (e.g. A-01).
  5. Add items — Items view → NEW ITEM, assign each product to a bay.
  6. Pair robots — Fleet → PAIR ROBOT (see section 6).
  7. Assign tasks — Tasks view or a robot's page when work is ready.

Set your organization name in Settings → General so the header shows your company name.

3 · Reading the board

The home page has four views — switch with the tabs above the card grid:

  • ITEMS — inventory tags for every product on file.
  • FLEET — robots grouped by home section; live online/offline status.
  • TASKS — work orders assigned to robots.
  • MAP — floor plan of the selected warehouse (zones, bays, robots).

The navigation deck filters everything below it:

  • Warehouse — click to focus on one site.
  • Aisle / zone — the walkway or storage zone (e.g. Aisle A, Receiving). This is a section in the system.
  • Bays — shelf positions inside that aisle; the black number is how many items sit on that bay. ALL BAYS clears the bay filter.

The counters row shows live totals: warehouses, bays, items, robots, and open tasks. Use the FIND search box to look up items (by name or SKU) or filter tasks/fleet by keyword.

Top right: your account menu (Settings, Staff guide, Log out) and ALERTS (live notification feed).

4 · Items

Each item appears as a tag: name, yellow SKU stamp, barcode strip, and location chips at the bottom.

  • Add — Items view → first NEW ITEM card → name, optional SKU, shelf, notes → Save.
  • Edit or move — click any tag. Change the shelf to move it to another bay.
  • Detail page — click a tag to open its full record at /items/<id>.
  • Delete — open the item → DELETE.
  • Search — FIND box filters by name or SKU across the warehouse.

5 · Fleet & robots

Robots are workers, not inventory. Switch to FLEET to monitor them. Robots are grouped by home section — where they are stationed when idle. Unassigned robots appear in their own group.

Each robot shows a coloured status dot. Status is live — reported by the robot over the network. Staff cannot set online/offline manually.

  • Working — carrying out a queued or in-progress task.
  • Idle — connected and available.
  • Charging — docked; not available for work.
  • Offline — not connected; no heartbeat for ~10 seconds.

The fleet and tasks views auto-refresh every few seconds so online/offline changes appear without reloading the page. The map view does the same.

Click any robot card to open its detail page. The first card is PAIR ROBOT for registering new hardware.

6 · Pairing an ESP32 robot

Pairing links a physical robot (ESP32 with OLED screen) to a record in WarehouseDB. Each robot has a unique device ID (its Wi-Fi MAC address) and shows a 6-digit pairing code on its screen.

Step by step

  1. Power on the robot. It connects to Wi-Fi and shows a 6-digit code on the OLED (e.g. 695577).
  2. On the board: FLEET → PAIR ROBOT (or Settings → Fleet).
  3. Enter the code, a display name, and optionally a home section.
  4. Click START PAIRING. The app waits for the robot to confirm.
  5. Within a few seconds the robot calls the server and both sides show connected.

Already paired / duplicate protection

  • If you enter a code that is already paired, the app shows an error under the code field with a link to the existing robot — you cannot register it twice.
  • If pairing is in progress for that code, wait for the robot to connect or remove the pending slot.
  • If the same physical robot tries to pair again, the ESP32 screen shows ALREADY PAIRED or recovers automatically if it is the same device.

Network requirements

  • The server PC and the ESP32 must be on the same Wi-Fi network (2.4 GHz).
  • The server must listen on 0.0.0.0:8000 (not only localhost) so the robot can reach it.
  • Use your PC's LAN IP in the robot's config.h (e.g. 192.168.4.21), not 127.0.0.1.

After Clear all data or deleting a robot from the fleet, the ESP32 detects disconnection within ~1 second and returns to pairing mode with its code on screen. You must pair again.

7 · Robot detail page

Click a robot in Fleet or Map to open /robots/<id>. This page has three areas:

  • Robot panel — edit display name and home section (base on the map). Live status, paired time, and last-seen time update automatically. Delete removes the robot from the fleet.
  • Home base map — mini floor plan showing which section this robot belongs to.
  • Assigned tasks — list of work orders with #ID badges, status dropdowns, and delete buttons.

Home section is where the robot lives when idle. Task destination (Go to section) is where a specific job sends it — these can be different. Use USE HOME when assigning a task only if the work is at its base.

8 · Tasks & work order IDs

Switch to TASKS to assign and track work. A task tells one robot what to do and where.

  • Assign — NEW TASK → pick robot, action (pick / restock / move / inspect / charge), destination section (required), optional item and note → Save.
  • Work order ID — every task gets a unique number (e.g. #8). This appears on task cards, the robot page, the assign toast, and the ESP32 OLED screen when the robot picks up the job.
  • Statuses — Queued → In progress → Done (or Cancelled). Click a task card to edit.
  • The Open tasks counter totals everything queued or in progress.

When a robot has an active task, its screen shows the ID and action (e.g. #8 / PICK / ONLINE).

9 · Map & floor plan

Switch to MAP for a live floor plan of the selected warehouse. The view includes:

  • A toolbar with zone count, bay count, item count, and robots online.
  • A status legend (Working · Idle · Charging · Offline).
  • One zone card per section — bays with stock counts, robots stationed there.
  • A Dock card for unassigned robots (hatched pattern).

Zone cards with robots show a yellow top stripe. Click a robot pill to open its detail page. The map auto-refreshes every 5 seconds while you are on this tab.

10 · Alerts & live feed

Click ALERTS (bell icon, top right) to open the notification panel. Events appear in real time:

  • New tasks created
  • Pick requests — when you ask for items from the garage shop
  • Robot pairing started or completed
  • Robot connected / disconnected activity

Inside the panel:

  • MARK ALL READ — clears the unread badge without deleting history.
  • CLEAR ALL — removes every alert from the list.
  • Click an alert to mark it read and jump to the related robot or task.

The green dot on the ALERTS button pulses when the live server feed is connected.

11 · Settings

Open Settings from the account menu. Panels:

  • General — organization name in the header.
  • Account — change username and password (current password required).
  • Appearance — rename and recolor the four robot status labels.
  • Fleet — list all robots; pair new ones with a code.
  • Locations — tree of warehouses, sections, bays, and item counts; add warehouses.
  • Data — export, import, and clear (see section 13).
  • Security — login lockout, session timeout, IP allowlist (see section 12).
  • System — live database statistics.
  • Integration — HTTP endpoints for robot firmware and external tools.

You can also add warehouses, sections, and bays from the ＋ chips on the main deck. Deleting a location removes its bays and items. Robots in a deleted section become Unassigned, not deleted.

12 · Security

Settings → Security controls how staff access the board:

  • Max failed logins — after this many wrong passwords, the account is locked temporarily.
  • Lockout duration — how long the lock lasts (minutes).
  • Session timeout — hours before an idle login expires.
  • IP allowlist — optional. Restricts staff browsers to specific IPs or subnets. Robots on your LAN can always reach pairing, heartbeat, and task endpoints regardless of this setting.

Export, import, and clear-all-data actions require your current password. Password changes must meet the minimum policy (length + letter + number).

If you enable the IP allowlist, add your office subnet (e.g. 192.168.4.0/24). Using only 127.0.0.1 will block other computers on the network from opening the board.

13 · Backup, import & clear

Settings → Data:

  • Export JSON backup — downloads warehouses, items, robots (without pairing codes), and settings. Tasks and alerts are server-only and are not included.
  • Import JSON backup — replaces all warehouse data from a file. Requires password. Robots may need to pair again (codes are not in backups).
  • Clear all data — deletes warehouses, items, robots, tasks, and alerts. Keeps your login and settings. Requires password. ESP32 units show DISCONNECTED and must be paired again.

Database file on the server: instance/warehouse.db (SQLite).

14 · Physical robot (ESP32)

The ESP32 firmware lives under Arduino/WarehouseDB_<unitId>/ — one folder per physical robot (e.g. WarehouseDB_55EF4). No display names in the folder; name the robot when you pair in the app.

What the screen shows

  • Pairing mode — 6-digit code, “enter code in app”.
  • Connected — robot name, ONLINE, idle or task info.
  • Active task — #8, action name (e.g. PICK), ONLINE.
  • Disconnected — if removed from the app or database cleared: DISCONNECTED + pairing code.
  • Offline — server unreachable but still paired; retries automatically.

How it talks to the server

  • Heartbeat — every 1 second (POST /api/robots/<id>/heartbeat). Keeps the robot online in the app.
  • Task poll — every 5 seconds (GET /api/robots/<id>/tasks). Reads queued work and shows the task ID on screen.
  • Both require header X-Robot-Code: <6 digits>.

Re-upload the Arduino sketch after firmware updates. Bump FIRMWARE_VERSION in config.h when you ship a release — the robot page compares it to Arduino/firmware.json on the server. Edit Wi-Fi SSID, password, and server LAN IP in config.h before the first upload.

15 · Robot API reference

Robots and integrations use HTTP on the same server as the board. Key endpoints:

GET  /api/robots/ping              — health check (no auth)
POST /api/robots/claim              — robot confirms pairing (body: pairing_code, device_id)
POST /api/robots/pair               — staff links a code to a name (session login)
GET  /api/robots/<id>/pair-status   — poll while waiting for robot
POST /api/robots/<id>/heartbeat     — robot status ping (header: X-Robot-Code)
GET  /api/robots/<id>/tasks        — robot reads assigned work (header: X-Robot-Code)
GET  /api/items/<id>                — resolve item → location path
GET  /api/tree                      — full warehouse hierarchy

Item lookup example — returns location.path ready for navigation:

GET http://localhost:8000/api/items/42

Full technical details: Settings → Integration, and README.md in the project folder.
